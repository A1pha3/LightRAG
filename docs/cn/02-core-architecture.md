# LightRAG 核心概念：架构与查询模式深度解析

> 💡 **学习目标**：
> - 掌握 LightRAG 的四大存储支柱架构及其各自的职责。
> - 深入理解 5 种查询模式的区别，并能根据业务场景选择最合适的模式。

## 1. 四大存储支柱（Storage Layer）

为了保证极高的扩展性和性能，LightRAG 将存储层解耦为四个独立的部分。每次操作都会与这些存储层交互：

1. **KV 存储（KV_STORAGE）**
   - **作用**：存储 LLM 的响应缓存、切分后的文本块（Chunks）以及文档元数据。
   - **为什么需要它**：缓存 LLM 响应可以极大地节省重复查询的 API 成本；存储文本块是进行最后阶段上下文拼接的基础。
2. **向量存储（VECTOR_STORAGE）**
   - **作用**：存储实体、关系和文本块的高维 Embedding 向量。
   - **为什么需要它**：当用户提问时，系统需要通过计算余弦相似度，快速在大规模数据中找到语义最相关的图节点和文本。
3. **图存储（GRAPH_STORAGE）**
   - **作用**：持久化保存由 LLM 提取出来的“实体-关系”图结构。
   - **为什么需要它**：图结构是 LightRAG 的灵魂。通过图遍历，系统能够发现文档中相隔甚远但逻辑相连的信息。
4. **文档状态存储（DOC_STATUS_STORAGE）**
   - **作用**：跟踪和记录每个文档的处理进度（如：是否已解析、是否发生错误）。
   - **为什么需要它**：在批量处理成千上万篇文档时，支持断点续传和增量更新。

> 📌 **隔离机制**：通过在实例化时传入 `workspace` 参数，可以在同一个数据库中实现多租户的数据隔离。

## 2. 深入理解查询模式（Query Modes）

LightRAG 提供了 `QueryParam(mode="...")` 来控制检索的策略。理解它们是调优检索质量的关键。

### 2.1 naive（基础模式）
- **原理**：直接进行向量相似度搜索，完全不使用知识图谱。
- **适用场景**：简单的事实查找，或者对比基线性能。速度最快，但缺乏深度。

### 2.2 local（局部模式）
- **原理**：从用户问题中提取关键实体，然后在图谱中找到这些实体，并提取它们的**直接邻居**和相关关系。
- **适用场景**：需要深入挖掘某个特定实体细节的场景（例如：“史蒂夫·乔布斯在 2007 年发布了什么产品？”）。

### 2.3 global（全局模式）
- **原理**：利用图谱社区（Community）的层级结构。它检索整个图谱的高层摘要信息，而不是拘泥于单个节点。
- **适用场景**：需要宏观总结和趋势分析的场景（例如：“这篇财报反映了公司怎样的战略转型？”）。

### 2.4 hybrid（混合模式）
- **原理**：并行执行 `local` 和 `global` 检索，将微观细节与宏观上下文拼接在一起，交给 LLM 生成最终答案。
- **适用场景**：综合性问答，是日常使用中平衡性较好的默认选择。

### 2.5 mix（高阶融合模式 - 推荐）
- **原理**：深度整合了知识图谱结构检索与纯文本块的向量检索。如果配置了重排器（Reranker），它会对检索到的所有内容进行重新打分排序。
- **适用场景**：生产环境中的首选模式。结合 Reranker 使用时，能够提供最高质量的检索准确率。
